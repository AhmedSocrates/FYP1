> **Procedural World Generation via Fine-Tuned Embedded Scripting**

Welcome to the **Agentic 3D Engine**. This project is a minimal, research-driven 3D game engine designed from the ground up to be coded, controlled, and populated entirely by an autonomous Artificial Intelligence agent at runtime. 

Instead of treating Artificial Intelligence as a static asset generator, this system treats the AI as an active **Software Engineer**.

---

##  The Problem Context

The current landscape of 3D environment generation is split into two extremes:
1. **Traditional Engines (Unreal/Unity):** Highly deterministic and performant, but require massive amounts of manual human labor or complex visual scripting to place objects, configure lighting, and set up spatial logic.
2. **Pure Generative AI (Text-to-3D / NeRFs):** Models that output meshes or point clouds directly from text. These are computationally massive, act as uneditable "black boxes," and struggle fundamentally with precise spatial reasoning and collision logic.

##  The Solution: A Neuro-Symbolic Approach

This project bridges the gap by building a "sandbox" specifically for an AI. 

When a user provides a natural language prompt (e.g., *"Build a dark medieval camp with 3 tents arranged in a circle around a fire"*), a fine-tuned Large Language Model (LLM) instantly writes the scripting logic required to generate, place, and arrange the 3D environment. 

The engine instantly parses this generated code, rendering the procedural world in real-time.

---

##  Core Architecture

The system is built on a modular, three-layer architecture to separate the rendering mathematics from the AI logic:

### 1. The Rendering Core (Low-Level Engine)
Built in **C++ and OpenGL**, this is the foundation of the system. It handles the heavy computational lifting that LLMs are traditionally bad at:
* Window creation and Render loops.
* Shader compilation and mesh loading (`.obj`).
* Basic lighting paradigms and camera systems.
* Axis-Aligned Bounding Box (AABB) collision detection.

### 2. The Scripting Bridge (The API)
A deterministic embedded API (using **Python**) that allows external scripts to manipulate the engine state at runtime without requiring a C++ recompilation. It exposes a strict vocabulary of commands to the AI, such as:
* `world.set_time(time_of_day)`
* `world.spawn_entity(asset_id, x, y, z)`
* `physics.check_overlap(entity_a, entity_b)`

### 3. The Agent Developer (Semantic Processor)
A local, open-weight LLM fine-tuned specifically on the engine's API. It acts as a translator between human intent (natural language) and the engine's deterministic rule set (executable scripts).

---

##  The Secret Sauce: The Agentic Feedback Loop

LLMs naturally hallucinate and write buggy code. The core innovation of this engine is its **Agentic Feedback Loop**. 

If the AI agent writes a script with a syntax error, or if its spatial math places a tree *inside* a solid house, the system does not crash. Instead:
1. The C++ runtime catches the exception or collision logic error.
2. The engine converts the error into a structured text prompt (e.g., *"Syntax Error on line 14: 'spawn_tree' is not a recognized function."*).
3. The error is fed **back** to the LLM.
4. The AI autonomously debugs its own code and re-submits the script until the world generates successfully.

---

##  Academic Research Focus

Developed as a formal research project, this repository serves as a testing apparatus to measure the spatial reasoning and self-correction capabilities of smaller, locally hosted LLMs. The research actively evaluates:

* **Execution Success Rate (ESR):** The frequency at which the AI generates syntactically correct, executable code on the first attempt.
* **Self-Correction Rate (SCR):** The AI's ability to successfully debug its own spatial and logical errors via the engine's feedback loop.
* **Constraint Satisfaction:** The accuracy of the AI in adhering strictly to numerical and spatial instructions provided in the human prompt.
* **Spatial Coherence:** The frequency of logical collision errors (e.g., floating objects, overlapping meshes) generated by the model's spatial planning.
